# Agent Guide for Hyfata Music App

> This file is intended for AI coding agents. It contains essential project context, build instructions, and conventions needed to work effectively in this codebase.

## Project Overview

**Hyfata Music App** is a cross-platform music player application built with [Flutter](https://flutter.dev). According to the project's README, the app is intended to:

- Play music from local device storage.
- Stream music from YouTube.
- Control the [Hyfata Music Bot](https://github.com/Hyfata/Hyfata-MusicBot).

The project is currently in an early stage. The `lib/main.dart` file still contains the default Flutter counter-app template generated by `flutter create`, and no custom business logic or UI for music playback has been implemented yet.

- **Primary Language:** Dart
- **License:** GNU General Public License v3.0 (see `LICENSE`)
- **Supported UI Languages:** English, Korean

### Key Configuration Files

- **`pubspec.yaml`** — Declares the package name (`hyfata_music_app`), Dart SDK constraint (`^3.11.5`), dependencies, and Flutter-specific settings (e.g., `uses-material-design: true`).
- **`analysis_options.yaml`** — Includes `package:flutter_lints/flutter.yaml`, enforcing the standard set of recommended lints for Flutter apps.
- **`android/app/build.gradle.kts`** — Android build script. Uses `com.android.application`, Kotlin Android plugin, and the Flutter Gradle plugin. Application ID is `com.example.hyfata_music_app`.
- **`android/build.gradle.kts`** — Root Gradle project configuration.
- **`web/manifest.json`** — PWA manifest.

## Testing Instructions

Tests are located in the `test/` directory and use the `flutter_test` package.

```bash
# Run all tests
flutter test

# Run with coverage
flutter test --coverage
```

Currently, the only test file is `test/widget_test.dart`, which performs a basic smoke test on the default counter app. As the project evolves, new unit tests, widget tests, and integration tests should be added alongside the source code they validate.

## Code Style Guidelines

- **Linter:** The project follows the rules defined in `package:flutter_lints/flutter.yaml`. Do not disable lints globally without a strong reason; prefer inline suppression (`// ignore: rule_name`) when necessary.
- **Formatting:** Use the standard Dart formatter (`dart format` or `flutter format`).
- **Language:** All code comments and documentation should be written in **English**.
- **Widget Style:** Prefer `const` constructors where possible. Follow Flutter best practices for widget composition and state management.

## Security Considerations

- The Android release build currently signs with the **debug keystore** (`signingConfig = signingConfigs.getByName("debug")`). Before any production distribution, configure a proper release signing config in `android/app/build.gradle.kts`.
- There are no `.env` files or secret management configurations present yet. If API keys (e.g., for YouTube Data API or bot communication) are introduced in the future, they must **not** be committed to version control. Use platform-specific secret management or Dart environment-variable solutions.
- The app targets multiple platforms; any native plugin that handles network requests or file system access should be reviewed for platform-specific permission requirements.

## Deployment Notes

- **Versioning:** Controlled via `pubspec.yaml` (`version: 1.0.0+1`). The `+1` suffix is the build number used by Android (`versionCode`) and iOS (`CFBundleVersion`).
- **Web:** The `web/` directory is configured as a standard Flutter web app with a PWA manifest. The `base href` is set to `$FLUTTER_BASE_HREF` for flexible hosting paths.
- **Android:** Application ID is currently `com.example.hyfata_music_app`. Update this to a non-example domain before publishing.

## Conventions for Agents

- When adding new Dart source files, place them under `lib/`. Consider creating subdirectories (e.g., `lib/screens/`, `lib/models/`, `lib/services/`) as the codebase grows beyond the single-file template.
- When adding new tests, mirror the `lib/` structure under `test/` where practical.
- Do not commit generated files (e.g., `build/`, `.dart_tool/`, platform-specific ephemeral files). The `.gitignore` already covers standard Flutter/Dart generated artifacts.
- The `.metadata` file is managed by the Flutter tool. Do not edit it manually.
